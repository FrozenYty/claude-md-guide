# 7. 交叉引用规范

[English version](07-cross-reference-discipline.md)

## 为什么有这条规则

数字、章节编号、文件名和配置值是任何项目中被复制粘贴最多的字符串。它们出现在索引表、README、CHANGELOG、eval 描述、路由表以及内联注释中。当你修改源值时，每个副本都会悄悄地漂移——而且这些是代码审查中最难注意到的 bug，因为文本单独看完全合理。错误的章节编号下面不会出现红色波浪线。

一个具体案例：`drawio-templates.md` 被重新组织了——删除了四个模板，新增了十一个。实际的模板移到了新的章节（§1-§4 架构，§5-§15 布局/经典）。但文件顶部的索引表从未更新。**十个中有十个**索引条目指向了错误的章节。用户按照“流程图 → §3”操作，会跳转到 RAG 管线模板。

这个错误随后传播到了 `examples/README.md`（Diffusion 标记为 §5，实际是 §2）、`evals/evals.json`（同样的 §5→§2 错误）以及 `CHANGELOG.md`（声称 19 个模板，实际是 15 个；声称 8 个架构模板，实际是 4 个）。三个独立的审计 agent 发现了同一 bug 的不同碎片——没有一个意识到这是系统性的。

## 实践中是什么样子

**在修改一个出现在别处的值之前：**
```bash
grep -r "§5" .          # 找到所有对旧章节编号的引用
grep -r "19 templates" . # 找到所有对旧计数的引用
grep -r "8 architecture" . # 找到所有对旧分类计数的引用
```
修复每一个匹配项。如果旧值在多个文件中是错误的，那这是同一个 bug，而不是 N 个独立的 commit。

## 什么时候可以放宽

如果被修改的值只存在于一个文件中，不会出现在其他任何地方（例如单个脚本中的内部变量名），那么搜索步骤只是走过场。这条规则在某个值已经被文档化、被索引或被其他文件引用时激活——对于计数、章节编号和文件路径来说，几乎总是如此。
