# 5. 语言

[English version](05-language.md)

## 为什么有这条规则

这条规则有两个层面：（a）默认回复语言，以及（b）中文输出所要求的 Unicode 排版规范。

回复语言层面的存在，是因为在 agent 猜错语言时，双语用户会浪费时间翻译。默认使用中文、按需切换到英文，可以消除这种摩擦。

排版层面的存在，是因为 LLM 经常在中文字符旁边输出 ASCII 引号 `"`（U+0022）——这是中文 LLM 输出中最常见的格式错误。全角引号 `""`（U+201C/U+201D）是中文排版的强制性要求，但在大多数等宽编辑器中，它们与 ASCII `"` 在视觉上无法区分。你不能依赖肉眼来捕捉这个错误。

一个具体案例：在一次对 24 个 prompt 文件的审计中，有一个文件——`humanize-zh.md`，即*定义中文排版规则*的那个文件——存在五处不同的引号 bug。其中包括中文语境中的 ASCII 引号、一对方向颠倒的引号（`U+201D...U+201C` 而非 `U+201C...U+201D`），以及一个被用作右引号的左引号。人工校对者扫读这个文件时看不出任何差异。

## 实践中是什么样子

**坏——肉眼不可见：**
```
carries an obvious "machine flavor" or "translationese" into natural academic prose
```
`"machine flavor"` 和 `"translationese"` 周围的引号是 ASCII 引号 `"`（U+0022）。它们看起来正确，但在中文语境中排版上是错误的。

**好——在字节层面验证：**
```python
python -c "print([hex(ord(c)) for c in line if ord(c)>127])"
```
这个一行命令可以显示字符串中的每一个非 ASCII 字符。U+201C 和 U+201D 是正确的；U+0022 则不是。

## 什么时候可以放宽

当整个所在段落是英文时（例如代码注释或全英文的技术章节），可以使用 ASCII 引号。同样，当输出语言仅为英文时，中文排版规则不适用。这条规则在输出中出现中文字符时才激活。
